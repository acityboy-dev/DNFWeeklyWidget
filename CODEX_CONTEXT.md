# DNF Weekly Widget - Codex 작업 인계

이 문서는 다른 PC나 새 Codex 세션에서 프로젝트와 작업 맥락을 빠르게 이어받기 위한 기록이다.
장기적으로 유효한 결정, 작동 방식, 진행 상태, TODO가 생기면 필요한 시점에 갱신한다.

## 프로젝트 요약

DNF Weekly Widget은 던전앤파이터 여러 캐릭터의 주간 콘텐츠 진행 상태와 주간 획득 정보를 한 화면에서 확인하는 Windows용 WPF 위젯이다.

여러 캐릭터를 일일이 접속하거나 개별 조회하지 않고도 다음 정보를 빠르게 파악하는 것이 목적이다.

- 캐릭터별 명성, 직업 및 서버 정보
- 레기온과 레이드 등 선택한 주간 콘텐츠의 완료 여부
- 중천 장비, 서약, 결정 등 선택한 주간 획득 정보
- 미완료 콘텐츠가 남은 캐릭터와 전체 주간 현황 요약
- 주간 초기화까지 남은 시간과 정기점검 반영 시각

이 앱은 게임 클라이언트를 조작하는 도구가 아니다. Neople OpenAPI와 외부 조회 결과를 읽어 현황을 표시하는 개인용 관리 위젯이다.

## 주요 기능

- 서버와 캐릭터명을 입력해 캐릭터 카드를 추가한다.
- 던담의 모험단 검색 결과로 현재 프리셋의 캐릭터 목록을 한 번에 불러온다.
- 캐릭터 카드를 드래그해 순서를 변경하거나 삭제 영역으로 드롭해 제거한다.
- 카드 엔트리를 여러 프리셋으로 나누어 관리한다.
- 프리셋별 캐릭터 카드에 독립적인 메모를 작성하고 표시 여부를 설정한다.
- 상단 버튼으로 레이드 콘텐츠를 숨기거나, 목요일 06:00부터 토요일 06:00 전까지 자동으로 숨긴다.
- 미완료 캐릭터만 필터링하고, 카드 수와 표시 방식을 조절한다.
- 수동 갱신, 시작 시 자동 갱신, 5~180분 간격의 주기 갱신을 지원한다.
- 라이트/다크/시스템 테마, 저성능 모드, 컴팩트 모드, 캐릭터 이미지 표시 방식을 지원한다.
- 창을 닫으면 기본적으로 트레이로 숨고, 설정에 따라 작업 표시줄에도 표시한다.

## 기본 사용방법

1. Neople Developers에서 개인 API Key를 발급받는다.
2. 앱의 `설정 > 일반`에서 API Key를 입력하고 저장한다.
3. 메인 화면에서 서버와 캐릭터명을 선택하여 캐릭터를 추가한다.
4. 여러 캐릭터를 한꺼번에 넣으려면 모험단명을 입력해 던담에서 불러온다.
5. `지금 갱신` 버튼이나 `F5`로 최신 정보를 조회한다.
6. 필요하면 미완료 필터, 컴팩트 모드, 카드/줄 수와 주간 콘텐츠 표시 옵션을 조절한다.
7. 서로 다른 캐릭터 묶음은 카드 엔트리 프리셋으로 분리한다.

## 작동 로직

### 앱 시작

1. `%AppData%\DNFWeeklyWidget\settings.json`에서 설정, 프리셋, 캐릭터 목록과 카드 캐시를 읽는다.
2. 사용자 데이터 저장이 켜져 있으면 저장된 카드 캐시를 먼저 화면에 복원한다.
3. 주간 초기화 카운트다운과 정기점검 공지 확인 타이머를 시작한다.
4. 자동 갱신이 켜져 있으면 창 로드 후 Neople API로 최신 정보를 조회한다.

따라서 캐시가 있을 때는 네트워크 갱신이 끝나기 전에도 이전 조회 결과를 즉시 볼 수 있다.

### 캐릭터 조회와 갱신

각 캐릭터 갱신은 대략 다음 순서로 진행된다.

1. 서버 ID와 캐릭터명으로 Neople 캐릭터 검색 API를 호출한다.
2. 검색 결과의 첫 캐릭터 ID로 상세 정보를 조회한다.
3. 현재 주간 시작 시각부터 현재까지의 타임라인을 조회한다.
4. 타임라인을 주간 획득 정보와 콘텐츠 클리어 정보로 변환한다.
5. 설정에서 선택한 콘텐츠만 카드와 하단 요약에 반영한다.
6. 완성된 카드 정보를 현재 프리셋의 캐시에 저장한다.

현재 주간의 기본 시작점은 로컬 시간 기준 `목요일 06:00`이다. 타임라인 종료 시각은 미래가 되지 않도록 현재 시각 이하로 제한한다.

주간 콘텐츠는 정의된 타임라인 코드별로 조회하며, API가 코드별 요청에 `400 Bad Request`를 반환하면 전체 타임라인 조회로 대체한 뒤 필요한 항목만 필터링한다. 중복 이벤트는 고유 키 기준으로 제거한다.

### 주간 초기화와 점검 공지

- 기본 초기화 시각은 다음 목요일 06:00으로 계산한다.
- 넥슨 DNF 공지사항의 정기점검 공지를 주기적으로 확인한다.
- 해당 주의 점검 시작 시각을 찾으면 카운트다운에 반영한다.
- 공지 조회 실패는 앱 사용을 막지 않으며 기본 초기화 시각을 계속 사용한다.

### 프리셋

- 각 프리셋은 고유 ID, 이름, 캐릭터 목록, 카드 캐시, 카드별 메모를 가진다.
- 프리셋 전환 시 해당 프리셋의 메모리 행 또는 저장된 카드 캐시를 표시한다.
- 메모는 `CardMemos`에 `(ServerId, CharacterName)` 기준으로 카드 캐시와 분리해 직렬화한다. 같은 캐릭터가 여러 프리셋에 있어도 메모는 공유하지 않는다.
- 카드 순서 변경과 제거 결과는 현재 프리셋에만 반영된다.
- 마지막 프리셋은 삭제할 수 없다.

## 저장 및 캐시 방식

### 설정 파일

- 경로: `%AppData%\DNFWeeklyWidget\settings.json`
- 형식: 들여쓰기된 JSON
- 내용: API Key, 테마, 창 위치/크기, 갱신 옵션, 표시 옵션, 프리셋, 캐릭터 목록, 카드 캐시
- 저장 시 임시 파일을 먼저 만든 뒤 기존 파일을 교체한다.
- 기존 파일은 `settings.json.bak`으로 백업된다.
- JSON이 손상되어 읽지 못하면 `settings.json.broken`으로 복사하고 기본 설정으로 시작한다.

API Key는 현재 Windows 사용자 범위의 DPAPI로 암호화해 저장하며, 기존 평문 설정은 로드 시 암호화 형식으로 마이그레이션한다.

### 저장 큐와 디바운스

- 즉시 저장이 필요한 변경은 설정 스냅샷을 백그라운드 저장 큐에 넣는다.
- 창 이동과 카드 순서처럼 연속으로 발생하는 변경은 450ms 디바운스 후 저장한다.
- 저장 스냅샷은 컬렉션을 복제하므로 UI에서 이후 변경되더라도 대기 중인 저장 내용이 흔들리지 않는다.
- 앱을 실제 종료할 때는 대기 중인 저장을 반영하고 저장 큐가 끝날 때까지 기다린다.

### 카드 데이터 캐시

- 카드에 표시된 캐릭터 상세 정보, 주간 상태, 요약 줄, 이미지 경로 등을 프리셋별 `CachedCards`에 저장한다.
- 카드 캐시는 별도 파일이 아니라 `settings.json` 내부에 직렬화된다.
- 앱 시작 시 캐시를 먼저 표시하고, 이후 자동 또는 수동 갱신 결과로 교체한다.
- 캐시는 오프라인에서도 마지막 조회 상태를 보여주기 위한 것이며 최신 상태를 보장하지 않는다.

### 캐릭터 이미지 캐시

- 경로: `%AppData%\DNFWeeklyWidget\ImageCache`
- 파일명: `{serverId}_{characterId}.png`
- 이미지 저장이 성공하면 로컬 파일을 사용한다.
- 다운로드 실패 시 기존 로컬 이미지가 있으면 그것을 사용하고, 없으면 Neople 이미지 URL을 사용한다.
- 이미지 표시를 꺼도 기존 캐시가 있으면 내부 이미지 소스로 유지될 수 있다.

### `사용자 데이터 저장` 옵션

- 켜짐: 설정, 창 위치, 프리셋, 캐릭터 목록, 카드 데이터와 이미지가 저장된다.
- 꺼짐: 현재 실행 중 변경사항만 적용되며 일반 저장과 카드 복원이 중단된다.
- 옵션 자체를 끈 상태는 다음 실행에도 반영해야 하므로 설정창 저장 시 한 번은 강제 저장한다.
- 이미지가 이미 캐시되어 있어도 이 옵션을 끄면 새 이미지 파일은 저장하지 않고 원격 URL을 사용한다.

## 외부 의존성과 데이터 출처

- Neople OpenAPI: 캐릭터 검색, 상세 정보, 타임라인과 캐릭터 이미지
- 던담(`dundam.xyz`): 모험단명으로 캐릭터 목록 일괄 검색
- 넥슨 DNF 공지사항: 해당 주 정기점검 시작 시각 확인

던담과 넥슨 공지 HTML/API는 공식 계약된 내부 인터페이스가 아니므로 상대 사이트 변경 시 파싱이 깨질 수 있다.

## 주요 코드 구성

- `MainWindow.xaml(.cs)`: 메인 UI, 사용자 조작, 갱신, 필터, 드래그 정렬, 트레이, 주간 카운트다운
- `SettingsWindow.xaml(.cs)`: 설정 UI와 설정값 반환, 일부 옵션 실시간 미리보기
- `AppSettings.cs`: 설정 모델, JSON 로드/저장, 저장 경로와 프리셋 기본값
- `SettingsPersistenceService.cs`: 스냅샷 생성, 디바운스와 순차 저장 큐
- `CharacterCardService.cs`: 캐릭터별 API 조회와 이미지 캐시 조립
- `NeopleApiClient.cs`: Neople API 및 타임라인 호출
- `DundamClient.cs`: 던담 모험단 캐릭터 목록 조회
- `PresetService.cs`: 프리셋별 캐릭터, 카드 캐시와 화면 행 관리
- `WeeklyResetNoticeService.cs`: 정기점검 공지 검색과 점검 시작 시각 파싱
- `CharacterModels.cs`: 카드, 캐시, 주간 콘텐츠 판정과 화면 요약 모델

## 현재 작업

카드별 메모, 레이드 미표시, 헤더 프리셋명 표시 기능을 구현했다.

- 메모 편집은 한글 IME 호환을 위해 메인 창 오버레이 입력창을 사용한다.
- 빈 메모는 카드에 영역을 만들지 않으며, 설정에서 전체 메모 표시를 끌 수 있다.
- 자동 레이드 미표시가 켜지면 수동 토글을 숨기고 목요일 06:00 이상 토요일 06:00 미만에만 레이드를 숨긴다.
- 메인 제목 옆에 현재 프리셋명을 작은 글씨와 말줄임으로 표시한다.
- 관련 자동 테스트 15개가 통과했다.

- `일반`: API Key, 자동 갱신, 작업 표시줄, 사용자 데이터 저장
- `화면`: 테마, 저성능 모드, 캐릭터 이미지, 카드/줄
- `주간 콘텐츠`: 주간 획득 정보, 레기온과 레이드 표시 옵션

기존 컨트롤 이름과 이벤트 연결은 유지하여 `SettingsWindow.xaml.cs`의 저장 및 미리보기 로직은 변경하지 않았다.

### 신규 설치 기본 설정

- 캐릭터 이미지: `컴팩트`
- 카드/줄: `4`
- 레기온: 베누스 꺼짐, 아포칼립스 켜짐
- 레이드: 바칼/하드 안개신/일반 나벨 꺼짐, 하드 나벨/이내 황혼전/디레지에 켜짐
- 이 기본값은 새 설정 생성 시 적용하며 기존 사용자의 저장값은 덮어쓰지 않는다.

## 리사이즈 성능 최적화

- 카드 폭 변경은 Dispatcher 렌더 큐에서 프레임당 최대 한 번 처리한다.
- 카드의 320/340px 파생 레이아웃 속성은 실제 임계점 통과 시에만 변경 알림을 보낸다.
- `FixedColumnWrapPanel`은 화면 주변 행만 생성하는 가상화 패널로 변경했다.
- 카드별 Marquee 타이머를 공용 `CompositionTarget.Rendering` 루프로 통합하고 창 리사이즈 중에는 정지한다.
- 헤더 레이아웃 갱신은 120ms, 창 위치 저장은 180ms 디바운스를 적용한다.
- 좌우 모든 방향의 리사이즈에서 카드 폭을 렌더 프레임당 최대 한 번 갱신한다.
- Release 빌드는 성공했다. 실제 스크롤, 카드 드래그, 좌우 리사이즈 체감은 사용자가 직접 확인한다.

## 작업 시 주의사항

- 작업 시작 당시 `DNFWeeklyWidget.sln`에는 사용자 변경사항이 이미 있었다. 되돌리거나 덮어쓰지 않는다.
- `AppSettings.cs`, `NeopleApiClient.cs`, `WeeklyResetNoticeService.cs` 일부 한글 문자열이 현재 콘솔에서 깨져 보인 적이 있다. 실제 파일 인코딩과 문자열 손상 여부를 별도로 확인하기 전에는 무관한 교정을 하지 않는다.
- 이 문서에는 다음 세션의 판단에 필요한 내용만 남기고, 완료되었거나 무효가 된 기록은 정리한다.

- 캐시 파일 크기 관리와 사용하지 않는 캐릭터 이미지 정리 기능을 검토한다.
- 던담 및 공지사항 파서 변경 감지를 위한 테스트나 실패 상태 표시를 검토한다.
- 네트워크 요청을 캐릭터별 순차 실행 중이므로, 캐릭터가 많을 때 제한된 동시성으로 갱신 속도를 개선할 수 있는지 검토한다.
- 탭 선택 상태를 다음 설정창 실행에도 유지할지는 현재 요구사항에 없으므로 보류한다.

## 자동 업데이트 구조

- Release 빌드만 시작 시 `DNFWeeklyWidget.Release` 저장소의 `update.json`을 확인한다.
- Visual Studio Debug 빌드는 컴파일 조건으로 자동 업데이트를 완전히 비활성화한다.
- 패키지는 SHA-256 검증 후 다운로드하며, 임시 위치의 독립 `update.exe`가 앱 종료 후 파일을 교체하고 재실행한다.
- `update.exe`는 .NET이 아닌 C++ Win32 정적 링크 단일 앱이다. ZIP을 별도 staging 폴더에 풀지 않고 설치 대상 옆의 `.update-new` 파일로 직접 추출한 뒤 교체한다.
- 배포 계약과 절차는 `UPDATE_RELEASE.md` 및 `scripts/Publish-Release.ps1`을 따른다.
- 0.1.3부터 메인 앱은 업데이트 매니페스트만 확인하며, 다운로드·검증·교체 진행률 UI는 `update.exe`가 담당한다. 시작 시 자동 확인은 설정에서 끌 수 있고 설정창의 수동 업데이트는 확인창을 생략한다.
